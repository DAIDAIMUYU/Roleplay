# 角色酒馆 V4 产品级完整需求开发报表（给用户阅读版）

## 1. 这次重做的核心结论

你现在要做的不是“把 AI 聊天功能补全”，而是重做一个真正可以长期维护的产品。旧版本的问题不是少了几个按钮，而是产品边界没有切开：访客、正式用户、站主、管理员、Demo、真实聊天、数据管理、上下文调试全部混在一起，AI 写代码时自然会乱堆 UI 和假数据。

V4 的核心定位是：

> 公开给别人看的，是一个不烧钱、不泄露数据的 Demo 酒馆大厅；真正给用户使用的，是一个登录后使用自己 API 的私人角色扮演工作台。

因此，接下来所有设计都围绕四个底层原则：

1. 访客 Demo 零持久化：不登录、不写数据库、不调用真实模型。
2. 真实用户登录后 BYOK：使用自己的 API Key，先做会话级临时使用。
3. 站主 Owner 私钥只在服务端使用，不给访客和普通用户消耗。
4. 先做上下文透明，再做复杂记忆和世界书。

## 2. 最终产品长什么样

最终产品不是一个页面，而是五个空间。

### 2.1 Hall 酒馆大厅

这是公开入口。访客打开网站先看到这里。

它负责说明：

- 这是一个角色扮演 AI 工作台。
- 当前是 Demo 模式。
- Demo 不是真实 AI，不消耗站主 API。
- 登录后可以连接自己的 API 使用。
- 展示几个公开 Demo 角色和几轮模拟聊天。

### 2.2 Chat Room 聊天房间

这是日常使用主场景。它的核心不是展示所有管理功能，而是让聊天舒服、可控、可解释。

桌面端：左侧角色/会话，中间聊天，右侧上下文控制台。

移动端：像 App 一样，聊天流优先，底部固定输入框，高级设置通过底部抽屉打开。

### 2.3 Studio 创作工坊

这里管理角色卡、世界书、记忆、提示词模板、模型预设。

之前 UI 乱的原因之一，就是把这些管理功能全部摊在聊天页里。V4 必须把 Chat 和 Studio 分开。

### 2.4 Settings 设置中心

管理账号、Provider、API Key 策略、备份恢复、外观、PWA。

用户配置 API 时必须能测试连接，并清楚看到错误原因。

### 2.5 Admin 管理后台

只给站主/管理员。它用于管理 Demo、系统模板、日志、成本、健康检查、审计。

管理员默认不应该看用户原始私聊。如果未来要排障查看，必须走 break-glass 审计流程。

## 3. 四种运行模式

| 模式 | 谁用 | 是否登录 | 是否写数据库 | 是否调用真实 AI | 说明 |
|---|---|---:|---:|---:|---|
| Demo | 访客 | 否 | 否 | 否 | 只看外观和模拟体验 |
| User BYOK | 普通用户 | 是 | 是 | 是 | 用自己的 API Key |
| Owner | 你自己 | 是 | 是 | 是 | 可使用服务端私钥 |
| Admin | 管理员 | 是 | 是 | 视情况 | 运维和审计 |

这四种模式必须从数据库、前端状态、API 路由、UI 文案上全部区分清楚。

## 4. 为什么访客不建议默认匿名登录

Supabase Anonymous Sign-In 技术上可行，但它会创建 authenticated 用户，并进入真实权限体系。你的访客目标只是看 Demo，不需要保存数据，也不应该产生真实用户记录和数据库写入。

所以 V4 的决策是：

- 访客默认 Demo Mode。
- 不创建 Supabase 用户。
- 不进入 RLS 真实数据面。
- 不调用真实 Provider。
- 只用 Mock Provider 或预设脚本。

这会极大降低安全和成本风险。

## 5. API Key 策略

你明确说过：访客不能用你的 API，真实用户想用就自己接 API。

所以第一版最稳的方案是：

- 用户登录后粘贴自己的 API Key。
- 只在当前会话/当前浏览器运行时使用。
- 暂时不长期保存到数据库。
- 通过 Provider Gateway 发起请求。
- 错误提示翻译成人话。

后期才考虑“加密托管 API Key”。这个功能一旦做，就意味着你要承担密钥保存责任，不建议早做。

## 6. 数据库主线

核心表包括：

- profiles
- user_roles
- model_presets
- characters
- character_revisions
- sessions
- session_participants
- branches
- messages
- prompt_templates
- worldbooks
- worldbook_entries
- memories
- context_runs
- trash_items
- backup_artifacts
- audit_events

核心原则：

- 所有私有数据都有 user_id。
- RLS 默认拒绝。
- 用户只读写自己的数据。
- Demo 数据只读且独立。
- 关键数据优先软删除或回收站。
- 备份和导出有 schema_version。

## 7. 上下文引擎是产品核心

角色扮演项目真正难的不是“把消息发给模型”，而是让模型稳定知道它该知道的内容。

每次请求应该由 contextBuilder 构建：

```text
系统规则
+ 角色卡核心设定
+ 当前用户 persona
+ 当前场景
+ 剧情摘要
+ 命中的世界书
+ 注入的记忆
+ 最近聊天记录
+ 当前用户输入
```

然后预算器负责裁剪，并告诉用户哪些内容被带入、哪些内容被裁掉。

这就是 Context Preview。它能解决用户最痛的几个问题：

- 为什么角色忘了？
- 为什么世界书没触发？
- 为什么回复变短？
- 为什么成本变高？
- 这次模型到底读了什么？

## 8. UI 必须重新按产品空间设计

旧版问题是像调试页，不像产品。V4 要按空间分层：

- Hall：给访客看懂。
- Chat：给用户沉浸聊天。
- Studio：给用户管理设定。
- Settings：给用户管理账号/API/数据。
- Admin：给站主管理系统。

移动端必须 App First：

- 底部 Tab。
- 聊天页底部输入框。
- 高级设置用 Bottom Sheet。
- 键盘弹出时输入框不能被遮挡。
- 字体和按钮要适合单手操作。

参考图只作为风格参考，不要求 100% 复刻。真正优先的是交互正确性、信息层级、安全边界和移动端可用性。

## 9. 阶段路线

### 阶段 0：项目清理、文档保护与干净底座

只做底座，不做业务。

### 阶段 1：运行模式 + Auth + 数据库 + RLS + Repository

把用户、权限、数据边界定死。

### 阶段 2：产品信息架构 + UI App Shell

做 Hall / Chat / Studio / Settings / Admin 的壳。

### 阶段 3：Provider Gateway + BYOK + Demo Mock

做 Mock Provider、BYOK 测试连接和统一网关。

### 阶段 4：核心聊天闭环

做真实聊天、流式输出、消息保存、重试、编辑重发、成本显示。

### 阶段 5：角色卡 / 提示词 / 会话管理

做角色、模板、会话、角色版本和绑定。

### 阶段 6：世界书 / 记忆 / 上下文引擎

做世界书触发、记忆审核、剧情摘要、Context Preview。

### 阶段 7：多角色群聊 / 分支剧情 / 备份恢复

做分支、群聊、回收站、导入导出和完整备份。

### 阶段 8：公开分享 / 管理后台 / PWA APP 预备

做 Admin、PWA、公开分享、健康检查和上线门禁。

## 10. 最重要的批评和风险

最危险的不是功能做不出来，而是 AI 再次乱做。

必须防止：

- 它删除需求文档。
- 它跳阶段。
- 它用 mock 假装真实功能完成。
- 它把 API Key 写前端。
- 它让访客调用真实模型。
- 它把 Demo 数据和私有数据混在一起。
- 它把世界书、记忆、角色卡、设置全部摊在聊天页。
- 它没有 RLS 测试就说数据库完成。

所以必须设置 AI 铁律：改代码前先 plan，删除文件先确认，每阶段必须验收，每阶段必须更新文档。

## 11. 最终建议

你应该按 V4 包重做，而不是继续修旧版。

最先执行的不是 UI，也不是聊天，而是阶段 0 和阶段 1。只要这两个阶段打牢，后面 UI、聊天、世界书、记忆、分支、PWA 都会变成在主干上生长的功能，而不是每次都推翻重做。

# V4.1 补充：Owner / Admin 与后期建议纳入情况

本次补丁确认两点：

1. 站主 Owner 与管理员 Admin 同时保留。
2. 前期提出的“后期可做功能”已经全部纳入阶段路线，不再作为散落建议遗漏在文档外。

## Owner 与 Admin 的区别

Owner 是系统最高拥有者，拥有最高权限；Admin 是站主授权的运营管理者，负责 Demo、模板、公告、反馈、基础运营等内容。早期可以由站主本人同时持有 Owner 与 Admin 两个账号，后期项目开放后再把 Admin 交给运营人员。

Admin 默认不能查看普通用户私人聊天，不能读取用户 API Key，不能修改服务端密钥、数据库结构和部署配置。这样后期即使把 Admin 账号交给别人，也不会破坏核心安全边界。

## 阶段完成后的最终形态

所有阶段完成后，系统应形成完整产品闭环：访客看 Demo，普通用户登录并使用自己的 API，Owner 管理系统最高权限，Admin 管理运营内容，移动端像 App 一样使用，桌面端适合创作管理，长篇 RP 通过世界书、记忆、上下文调试器和备份机制保持稳定。
