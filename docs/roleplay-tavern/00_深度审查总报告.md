# 00_深度审查总报告

## 执行摘要

本次 V4 需求包的核心判断是：角色酒馆不应该继续被当成“功能清单”开发，而应该被当成一个真正可发布、可维护、可扩展的产品来重做。

产品最终应被拆成两个清晰层面：

1. **公开 Demo 酒馆大厅**：面向访客，只展示外观、风格、交互气质和模拟体验；不登录、不写正式数据库、不调用真实 AI、不消耗站主 API。
2. **私人角色扮演工作台**：面向登录用户，支持 BYOK、自有数据隔离、角色卡、会话、世界书、记忆、上下文预览、备份恢复、移动端 App 化体验。

最关键的四个产品决策：

1. 访客模式 = 零持久化 Demo，不走真实模型。
2. 真实用户模式 = 登录后 BYOK，统一经过 Provider Gateway。
3. BYOK 首期 = 会话级临时使用，不急着长期存 Key。
4. 先做上下文透明，再做复杂记忆系统。

如果这四点不提前定死，项目很容易再次变成“UI 像产品、内部像临时脚本”的状态。

## 多角色审查结论

| 视角 | 直击痛点 | 必须补强 |
|---|---|---|
| CEO | 商业路径不清，容易做成大而全实验站 | Demo 转化漏斗 + BYOK 私有工作流 |
| 产品经理 | 功能堆叠，阶段边界不清 | 每阶段范围/非范围/验收/回滚 |
| 设计师 | UI 像功能面板，不像产品 | Hall / Chat / Studio / Settings / Admin 分区 |
| 前端 | 状态混乱、移动端易崩 | App First、Bottom Sheet、VisualViewport |
| 后端 | 权限、Provider、上下文混在一起 | Auth/RLS/Gateway/Repository 分层 |
| 测试 | 真模型不可稳定回归 | Mock Provider + context 快照测试 |
| 运维 | 成本、日志、备份没有产品化 | usage/context_runs/audit_events/备份矩阵 |
| 安全 | API Key、RLS、Admin 权限风险 | BYOK 临时策略、默认拒绝、break-glass |
| 用户 | 角色失忆、设定漂移、成本不透明 | Context Preview、成本卡、世界书命中解释 |
| 访客 | 不应看到你的私有数据或烧你的钱 | Demo Mode 完全隔离 |
| 管理员 | 运维需要能力，但不能窥探私聊 | 聚合视图 + 审计 + 权限分级 |

## 最终产品形态

用户打开后首先进入 Hall，而不是复杂后台。Hall 告诉用户：这是什么、能做什么、当前是否 Demo、如何登录真实使用。

登录后进入 Onboarding，先完成 Provider 连接测试。真实聊天必须经过 Provider Gateway。日常使用主要在 Chat Room，设定管理在 Studio，账号/API/备份在 Settings，系统运维在 Admin。

核心页面：

- Hall：公开展示和 Demo。
- Chat Room：沉浸聊天、上下文控制、成本透明。
- Studio：角色、世界书、记忆、提示词、模型预设。
- Settings：账号、Provider、数据备份、外观、PWA。
- Admin：Demo 管理、日志、审计、健康、成本。

## 最重要的架构边界

- Demo 不是匿名用户，不写正式库。
- User BYOK 与 Owner API 完全隔离。
- 前端组件不直接调用 Provider。
- 组件不直接访问 Supabase，统一走 Repository。
- contextBuilder 是独立引擎，不写在聊天组件里。
- Admin 默认不能看原始私聊。
- 备份不是只备数据库，还包括 Storage、配置、函数和 Auth 设置。

## 阶段路线总览

1. 阶段 0：项目清理、文档保护与干净底座。
2. 阶段 1：运行模式 + Auth + 数据库 + RLS + Repository。
3. 阶段 2：产品信息架构 + UI App Shell。
4. 阶段 3：Provider Gateway + BYOK + Demo Mock。
5. 阶段 4：核心聊天闭环。
6. 阶段 5：角色卡 / 提示词 / 会话管理。
7. 阶段 6：世界书 / 记忆 / 上下文引擎。
8. 阶段 7：多角色群聊 / 分支剧情 / 备份恢复。
9. 阶段 8：公开分享 / 管理后台 / PWA APP 预备。
