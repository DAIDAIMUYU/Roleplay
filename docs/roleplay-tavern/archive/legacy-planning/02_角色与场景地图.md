# 02_角色与场景地图

## 目标用户分层

| 用户类型 | 核心诉求 | 主要入口 | 风险 |
|---|---|---|---|
| 访客 | 快速理解产品外观和体验 | Hall / Demo | 看到真实私有数据、消耗站主 API |
| 普通注册用户 | 使用自己的 API 进行角色扮演 | Login → BYOK → Chat | API 配置失败、数据隔离失败 |
| 长线 RP 用户 | 角色稳定、剧情连续、能备份 | Studio / Chat / Context | 失忆、设定漂移、误删 |
| 站主 Owner | 自用、管理 Demo、维护产品 | Owner Mode / Admin | 私钥泄露、权限混乱 |
| 管理员 | 运维、审计、故障排查 | Admin | 权限过大导致隐私问题 |

## 访客路径

```text
打开网站
→ 进入酒馆大厅
→ 看到 Demo 角色与样式展示
→ 点击试玩 Demo
→ Mock AI 回复，不写库、不调真实模型
→ 被提示：登录并连接自己的 API 后可真实使用
```

## 普通用户路径

```text
打开网站
→ 登录/注册
→ Onboarding：连接 Provider
→ 测试 API Key
→ 创建/导入角色
→ 创建会话
→ 开始聊天
→ 查看上下文预览与成本
→ 导出备份
```

## 长线 RP 用户路径

```text
恢复旧会话
→ 查看剧情摘要
→ 调整当前场景
→ 世界书命中
→ 记忆审核
→ 创建分支
→ 导出会话备份
```

## 站主路径

```text
Owner 登录
→ 管理 Demo 数据
→ 管理默认模板
→ 查看系统健康
→ 查看成本汇总
→ 处理异常日志
```

## 管理员边界

管理员默认不应读取用户原始私聊。需要排障时必须走 break-glass 流程：

1. 明确原因
2. 记录审计事件
3. 限定时间窗口
4. 只读必要字段
5. 操作后关闭权限

## V4.1：Owner / Admin 权限边界补充

### 为什么保留 Owner 与 Admin

本项目早期由站主本人同时管理 Owner 与 Admin 账号，但从产品架构上仍保留两个身份，原因是：

- Owner 是系统所有者，拥有最高控制权。
- Admin 是站主授权的运营管理者，后期可交给别人负责日常运营。
- 早期一个人管两个账号，可以提前验证权限边界，避免后期开放管理人员时重构权限系统。

### 四类身份最终定义

| 身份 | 中文名 | 核心定位 | 是否可进入后台 | 是否可看普通用户私聊 | 是否可读用户 API Key |
|---|---|---|---:|---:|---:|
| Guest | 访客 | 只看 Demo 与 Mock AI | 否 | 否 | 否 |
| User | 普通用户 | 使用自己的角色、会话、API | 否 | 仅自己 | 仅自己本地/授权范围 |
| Admin | 管理员 | 运营管理 Demo、模板、公告、反馈 | 是 | 默认否 | 否 |
| Owner | 站主 | 系统最高拥有者 | 是 | 最高权限/需审计 | 不应明文读取 |

### Admin 可以管理

- Demo 角色、Demo 会话、Demo 世界书、Demo 提示词模板。
- 系统默认模板、默认分类、默认演示内容。
- 系统公告、更新日志、用户反馈、错误日志。
- 基础用户状态，例如禁用异常账号，具体能力后期再做细分。

### Admin 默认不能管理

- 不能修改 Owner 账号。
- 不能分配或移除 Owner 权限。
- 不能读取普通用户私人聊天内容。
- 不能读取用户 API Key。
- 不能读取或修改服务端密钥、service role key、环境变量。
- 不能修改数据库结构、RLS 策略、部署配置。
- 不能绕过审计流程查看敏感数据。

### Owner 可以管理

- Admin 的全部运营权限。
- 分配/移除管理员。
- 系统级配置和功能开关。
- Demo 数据、系统模板、公告、用户管理。
- 安全策略、后台模块、审计策略、部署策略。

### 早期实现建议

早期实现中，后台可以先支持：

1. Owner 账号。
2. Admin 账号。
3. `canAccessAdminPanel = role in ('owner', 'admin')`。
4. `canManageAdmins = role === 'owner'`。
5. `canManageDemoData = role in ('owner', 'admin')`。
6. `canViewPrivateUserData = role === 'owner'`，并且必须写审计日志。

Admin 后期交给别人时，不需要再改数据库大结构，只需要通过 Owner 授权对应账号即可。
