# 05_认证权限与数据分区

## 四种运行模式

| 模式 | 是否登录 | 是否写库 | 是否真实模型 | 说明 |
|---|---:|---:|---:|---|
| Demo | 否 | 否 | 否 | 访客只看演示，Mock AI，不烧钱 |
| User BYOK | 是 | 是 | 是 | 用户登录后使用自己的 API |
| Owner | 是 | 是 | 是 | 站主自用，可使用服务端私钥 |
| Admin | 是 | 是 | 视情况 | 运维、审计、Demo 管理 |

## 关键决定

访客不使用 Supabase Anonymous Sign-In 作为默认方案。原因：匿名用户仍进入 authenticated 权限面，会增加 RLS 和权限区分复杂度。当前访客只需看 Demo，不需要持久化。

## 数据分区

| visibility / scope | 含义 | 示例 |
|---|---|---|
| private | 用户私有 | 角色、会话、记忆 |
| demo | 公开演示 | Demo 角色与脚本 |
| shared | 用户主动分享 | 分享角色卡、模板 |
| system | 系统只读模板 | 默认模型预设、默认提示词 |
| admin | 管理数据 | 审计、公告、健康检查 |

## RLS 原则

- 默认拒绝。
- 用户只读写 `user_id = auth.uid()` 的数据。
- 公开 demo 数据通过单独表/视图或 `visibility='demo'` 只读策略暴露。
- Admin/Owner 使用 `user_roles` 与 `has_role()` 控制。
- 管理员默认不能读取用户原始私聊内容。

## 用户角色

- user
- owner
- admin
- support_readonly，可选

## 高危操作

以下操作必须二次确认，并写入 audit_events：

- 导出全部数据
- 删除全部数据
- 永久删除回收站
- 修改 Owner Provider
- 查看敏感日志
- 使用 break-glass 查看必要的用户数据
- 修改 RLS / Provider / Admin 设置

## Admin 权限原则

Admin 是“运营管理者”，不是“数据库超级用户”。因此：

- Admin 默认管理 `demo/system/admin` 分区的数据。
- Admin 默认不读取 `private` 用户数据。
- Admin 处理用户反馈时，只能看必要的元数据或用户主动提交的内容。
- 如必须排障查看敏感数据，应走 break-glass 流程，写入 `audit_events`。

## Owner 权限原则

Owner 是系统拥有者，具备最高权限，但产品实现仍应遵循最小权限原则：

- Owner 可以授权 Admin。
- Owner 可以管理 Demo、系统模板、公告、用户与后台配置。
- Owner 对敏感数据的访问应有审计记录。
- Owner 也不应直接明文读取用户 API Key；如果后期做加密托管 Key，只能通过安全服务端流程使用，不直接展示明文。

## 权限判断建议

```ts
const canAccessAdminPanel = role === 'owner' || role === 'admin'
const canManageAdmins = role === 'owner'
const canManageDemoData = role === 'owner' || role === 'admin'
const canManageSystemTemplates = role === 'owner' || role === 'admin'
const canManageSystemSecrets = role === 'owner'
const canViewPrivateUserData = role === 'owner' // 且必须审计；Admin 默认禁止
```

## RLS 策略修订原则

- 私人业务表：`user_id = auth.uid()` 可读写；Owner 可最高权限访问；Admin 默认不可访问 private 数据。
- Demo/System 表或记录：Guest 通过只读接口看 Demo；Admin/Owner 可管理。
- user_roles：普通用户只能查看自己的角色；Admin 可查看角色信息；只有 Owner 可以授予/移除 Admin 或 Owner 权限。
- audit_events：Admin/Owner 可查看运营审计；敏感访问必须写入记录。
