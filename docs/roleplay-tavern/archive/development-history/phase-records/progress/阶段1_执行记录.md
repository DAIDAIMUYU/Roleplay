## 四、RLS 策略清单

所有策略已在 `0001_initial_schema_draft.sql` 中预写好：

### 敏感 private 表（Admin 默认不可读）
- `sessions`, `branches`, `session_participants`, `messages`
- `memories`, `context_runs`, `trash_items`, `backup_artifacts`
- `character_revisions`

策略原则：`user_id = auth.uid() OR public.is_owner()`

### 运营内容表（Admin 可管理 demo/system/admin 分区）
- `characters`, `model_presets`, `prompt_templates`
- `worldbooks`, `worldbook_entries`

策略原则：`user_id = auth.uid() OR visibility IN ('demo','system','shared')`（只读），Admin/Owner 可写入 demo/system/admin 分区

### 角色与审计
- `user_roles`：普通用户看自己，Owner 可写入
- `audit_events`：actor 看自己，Owner/Admin 可查看
- `profiles`：自己 + Owner/Admin 可读，自己可写

## 五、权限模型说明

```
Owner ─── 系统最高权限，管理 Admin，可访问所有数据（审计记录）
  │
Admin ─── 运营管理者，管理 demo/system/admin 内容
  │       禁止读取普通用户 private 数据
  │
User  ─── 管理自己的 private 数据
  │
Guest ─── 只看 Demo，不写库，不调真实 AI
```

**前端权限判断函数**（`src/features/auth/permission.ts`）：
- `canAccessAdminPanel(role)` — Owner/Admin
- `canManageAdmins(role)` — Owner only
- `canManageDemoData(role)` — Owner/Admin
- `canManageSystemTemplates(role)` — Owner/Admin
- `canManageSystemSecrets(role)` — Owner only
- `canViewPrivateUserData(role)` — Owner only（需审计）
- `canReadSensitivePrivateData(role)` — Owner only

## 七、环境变量说明

`.env.example` 当前内容：
```
VITE_SUPABASE_URL=
VITE_SUPABASE_ANON_KEY=
VITE_APP_MODE=development
```

**Supabase 初始化步骤**：
1. 创建 Supabase 项目（supabase.com 或自托管）
2. 获取 `Project URL` 和 `anon public key`
3. 填入 `.env.local`（不提交到 Git）
4. 在 Supabase SQL Editor 中执行 `supabase/migrations/0001_initial_schema_draft.sql`
5. 执行 `supabase/migrations/0002_v41_owner_admin_policy_patch.sql`（如有）
6. `npm run dev` 启动后，`/login` 页面会显示 Supabase 是否已配置

## 十、Demo Mode 策略

- Guest 不登录 → `AuthProvider` 自动进入 `guest_demo` 模式
- Supabase 未配置 → `guest_demo` 模式，UI 显示配置引导
- Supabase 已配置但未登录 → `guest_demo` 模式
- 登录后 → `authenticated` 模式
- Demo 数据使用前端 mock（阶段 3 实现），不写真实数据库
- Demo 不调用真实 AI（阶段 3 强制 Mock Provider）
