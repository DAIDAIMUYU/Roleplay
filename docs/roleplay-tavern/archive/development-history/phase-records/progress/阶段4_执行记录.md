## 四、消息流程

### 发送消息
1. 构造 `ChatMessage { role: "user", content: text }`
2. 加入本地 messages state（即时 UI 反馈）
3. 调用 `sendProviderStreamRequest(isDemo, config, messages, abortSignal)`
4. 逐 `ChatStreamChunk` 更新 AI 回复（流式显示）
5. 完成后自动保存到 Repository（已登录）或内存（Demo）

### 流式接收
- `useChatSession` 消费 `AsyncIterable<ChatStreamChunk>`
- 每个 chunk 的 `content` 追加到当前 AI 消息
- 中间状态：消息气泡显示闪烁光标

### 停止生成
- `AbortController.abort()` → fetch 取消 → `ReadableStream.read()` 抛异常
- 已生成的部分内容保留在消息中

### 重新生成
- 删除最后一条 AI 消息 → 用相同上下文重新调用 Provider

### 编辑重发
- 编辑用户消息文本 → 截断该消息之后的所有内容 → 重新发送
- UI：底部弹出一行编辑栏（桌面）/ 顶部编辑栏（移动端）

### 删除/复制消息
- 删除：从本地 messages state 移除
- 复制：`navigator.clipboard.writeText()`

## 六、BYOK 真实聊天

- 登录用户进入 → 自动检测已保存的 API Key（localStorage / sessionStorage）
- 无 Key → 底部 amber 横幅"请配置 API Key" + 链接到 Settings

- 流式真实 API 回复
- 完成后用户消息 + AI 回复保存到 `messages` 表
- 自动创建分支（如无）并关联消息

## 七、流式输出

- **Mock**：逐字输出，20-50ms/字延迟模拟
- **DeepSeek / OpenAI Compatible**：`fetch` + `ReadableStream` + SSE `data:` 行解析
- 支持 `AbortSignal` 取消（停止生成）
- 解析 `data: [DONE]` 结束
- Token 用量从 `usage` 字段提取
- 中间状态：消息气泡末尾显示闪烁光标

## 八、错误处理

| 错误场景 | 处理方式 |
|---|---|
| 未配置 API Key | 底部 amber 横幅 + 禁用输入框 + 链接到 Settings |
| API Key 错误 (401) | 错误翻译 → 显示红色错误栏 + 重试按钮 |
| 余额不足 (402) | 错误翻译 |
| 限流 (429) | 错误翻译 + 标记 retryable |
| 网络失败 | 错误翻译 + 重试按钮 |
| 超时 | 错误翻译 |
| 用户取消生成 | AbortController → 保留已生成内容 |
| 保存失败 | saveStatus = "error" 红色文字显示 |
| Provider 返回非 200 | SSE 流第一帧错误 → 翻译后显示 |
