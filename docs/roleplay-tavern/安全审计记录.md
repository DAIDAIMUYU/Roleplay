# 安全审计记录

- **审计时间**：2026-05-19
- **审计范围**：前端代码、数据流、存储敏感度、日志安全、XSS 风险
- **审计方法**：代码搜索 + 静态分析

## 检查项目与结果

### 1. dangerouslySetInnerHTML (XSS)

- **结果**：**未发现**。代码库中不存在 `dangerouslySetInnerHTML` 使用。
- **评级**：通过

### 2. Markdown / HTML 渲染

- **结果**：**未使用 Markdown 渲染器**。没有 marked、DOMPurify、remark 等依赖。
- **评级**：通过（无渲染则无注入风险）

### 3. 导入 JSON 安全

- **检查**：DataManagement 导入功能
- **结果**：导入逻辑解析 JSON 后将内容写入数据库（通过 Repository），不直接渲染 HTML。不校验嵌套 XSS payload，但无渲染管道可利用。
- **评级**：P2 — 建议后续增加导入 schema 校验

### 4. API Key 日志风险

- **检查**：所有 console.log / console.warn / console.error
- **结果**：日志均使用 `[组件名]` 前缀，只记录错误消息和元数据。未发现 API Key、Authorization header、encryption_iv 等敏感内容打印。
- **评级**：通过

### 5. localStorage 敏感数据

- **发现**：
  - `apiKeyStorage.ts`：API Key 明文存储（local_device 模式，设计如此）
  - `apiProviderConfigStorage.ts`：配置元数据（不含 API Key）
  - `hostedCredentialsService.ts`：托管凭据选择记录（不含 secret）
  - `syncMetadata.ts`：同步元数据（不含 API Key）
- **评级**：P1 — local_device 明文存储 API Key 是已知风险，已在 UI 文档中标明

### 6. CSP / 安全 Headers

- **检查**：无 vercel.json，无 CSP 配置
- **建议**：生产部署时添加基础安全 headers（通过部署平台配置）
- **评级**：P2 — 建议添加但不阻塞当前发布

### 7. Supabase RLS

- **检查**：migrations 包含 RLS 策略，Repository 层不绕过 RLS
- **结果**：Supabase 数据库操作通过 Auth 认证的 session 进行，不暴露 service_role key 到前端
- **评级**：通过

### 8. Provider 错误回显

- **检查**：错误提示不包含 API Key
- **结果**：Provider 错误通过 `translateError()` 标准化处理，原错误消息不直接展示给用户
- **评级**：通过
