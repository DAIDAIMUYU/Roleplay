# 99_AI开发铁律与阶段执行提示词

## 最高优先级铁律

1. 不允许删除 `docs/`、`supabase/migrations/`、`README.md`、`.env.example`。
2. 不允许在没有明确授权的情况下批量删除文件。
3. 删除超过 5 个文件前，必须先输出删除清单、删除理由、影响范围，并等待用户确认。
4. 每次改代码前必须先输出计划：修改哪些文件、为什么改、阶段目标是什么、哪些明确不做。
5. 没有数据库表，不允许写对应真实 CRUD 页面。
6. 没有 TypeScript 类型，不允许写业务组件。
7. 没有 Repository / Service 层，不允许组件直接访问 Supabase 或 Provider。
8. Mock 只能作为 Demo / 测试 / fallback，不允许伪装成真实功能完成。
9. API Key 永远不写入前端代码、Git 仓库、日志、错误信息或公开数据库字段。
10. 访客 Demo 永远不调用真实模型，不写正式数据库。
11. 每个阶段必须更新阶段验收清单、未完成项和变更记录。
12. 每个阶段必须能运行、能回滚、能解释。
13. 参考图只作为风格方向，不允许像素级复刻导致产品结构跑偏。

## 阶段执行纪律

每个阶段固定流程：

```text
Plan → Implement → Self-check → Docs update → Test → Commit suggestion → Summary
```

每阶段输出必须包含：

- 本阶段目标
- 本阶段非目标
- 新增文件
- 修改文件
- 删除文件，如有
- 自检结果
- 验收清单
- 仍未完成内容
- 下一阶段建议

# V4.3 补充铁律：阶段 0-10 最终路线

## 1. 不允许使用“未来功能池”作为交付逃避

V4.3 已经将所有重要功能映射到阶段 0-10。执行任务时不得把 `15_阶段化功能总清单_无未来池版.md` 中列出的 P0/P1/P2 功能再次标记为“未来再做”。

P3 功能属于阶段 10 可选生态扩展，可以不影响阶段 9 核心封版，但必须明确说明未进入核心产品范围。

## 2. 阶段执行顺序

必须按以下顺序执行：

0. 项目清理、文档保护与干净底座
1. 运行模式 + Auth + 数据库 + RLS + Repository
2. 产品信息架构 + UI App Shell
3. Provider Gateway + Demo Mock + BYOK 本地版
4. 核心聊天闭环
5. 角色卡 / 提示词 / 会话 / 标签 / 搜索基础
6. 世界书 / 记忆 / 上下文引擎 / 快捷指令
7. 多角色群聊 / 分支剧情 / 数据安全基础
8. Admin 后台 / 公开分享 / API Key 加密托管 / 运营能力
9. 产品完善与封版层
10. 可选生态与商业化扩展

## 3. API Key 明确边界

- 阶段 3：只允许临时保存和本设备保存，禁止上传服务器。
- 阶段 8：才允许实现加密托管 API Key 和多设备同步。
- 任何阶段都禁止明文存储 API Key。
- Admin 和 Owner 后台都不得直接显示用户完整 API Key。

## 4. 阶段完成标准

每个阶段必须满足：

- 本阶段验收清单全部勾选。
- 项目可运行。
- 相关文档已更新。
- 未完成事项明确记录。
- Git commit 已完成。

否则不得进入下一阶段。
